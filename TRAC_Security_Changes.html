Comet — Security Changes Reference Document

Prepared for Western University Technology Risk Assessment (TRAC)  |  April 2026

This document describes the security controls implemented in the Comet classroom participation platform. All changes were made prior to the pilot submission to address standard vendor security requirements, FIPPA obligations under Bill 194, and best-practice edtech security posture.

R1 CSRF Protection

What it is: Cross-Site Request Forgery is an attack where a malicious website tricks a logged-in professor's browser into silently performing actions — such as deleting a class or exporting student grades — without the professor's knowledge.

What was done: Flask-WTF CSRF protection was installed and enabled. Every request from the professor's browser now must include a hidden one-time cryptographic token. The server rejects any request that is missing or has an invalid token. Student-facing routes are correctly exempted — they authenticate using Bearer tokens (not cookies), which are immune to CSRF by design.

R2 Rate Limiting and Account Lockout

What it is: Without limits, an attacker can automatically try thousands of password combinations against an account in seconds (brute-force attack).

What was done:

Rate limiting: Login endpoints now allow a maximum of 5 attempts per minute per IP address. Exceeding this returns HTTP 429 (Too Many Requests).

Account lockout: After 5 cumulative failed password attempts on the same account, the account is locked.

Professor unlock path: The professor uses "Forgot Password" to receive a reset link by email. Completing the reset clears the lockout counter.

Student unlock path: The lockout affects password login only. A student can still authenticate by tapping their RFID student card — hardware authentication is unaffected. Alternatively, the professor can reset the student's password from the class management interface.

R3 Student PII Exposure Fixed

What it is: The endpoint that allows students to look up their account before setting a first password was returning full personal information — name, email address, and student number — to anyone who submitted a student number. No login was required. Since Western student numbers follow a predictable format, this could allow bulk enumeration of student data.

What was done: The response was reduced to only the student's first name (sufficient to confirm the right account was found). No email address, last name, or student number is returned. The endpoint is also rate-limited to 5 requests per minute.

R4 HTTP Security Headers

What it is: Browsers rely on specific HTTP headers to enforce important safety rules. Without them, common attacks such as clickjacking, MIME-type confusion, and information leakage are possible.

What was done: A server-side hook now adds the following headers to every response:

X-Frame-Options: DENY — prevents the application from being loaded inside an iframe on another website, blocking clickjacking attacks.

X-Content-Type-Options: nosniff — prevents browsers from guessing file types, blocking certain injection attacks.

Referrer-Policy: strict-origin-when-cross-origin — limits what URL information is shared when a user navigates away from the application.

Permissions-Policy — explicitly disables camera, microphone, and geolocation access for the application domain.

Strict-Transport-Security (production only) — instructs browsers to always use HTTPS when connecting to this domain, enforced for one year.

R5 Session Cookie Hardening

What it is: The professor's login session is tracked by a browser cookie. Without security flags, that cookie can be stolen via JavaScript injection or transmitted over unencrypted connections.

What was done: Three security flags are now set on all session cookies:

HttpOnly — the cookie cannot be read by JavaScript, blocking the most common method of session token theft via XSS.

SameSite=Lax — the cookie is not sent on cross-site requests, providing an additional layer of CSRF protection.

Secure (production only) — the cookie is only transmitted over HTTPS, never plain HTTP.

3-hour inactivity timeout — sessions automatically expire, preventing indefinite access from unattended or shared devices.

R6 Audit Logging

What it is: Without a log of who accessed what and when, it is impossible to investigate a potential data breach or unauthorized access event — a requirement under FIPPA incident response obligations.

What was done: Structured audit log entries (prefixed [audit]) are now written for the following events, each recording the relevant user ID and IP address where applicable:

Professor login — success and failure  |  Professor logout  |  Class session start and stop

Gradebook export  |  Student roster export  |  Student data deletion

Student login — success (with authentication method: RFID, password, or first-time setup) and failure

Logs are written to the application's standard output, which is captured automatically by Azure App Service and retained in its log stream.

R7 FIPPA Data Deletion (Right to Erasure)

What it is: Ontario's Freedom of Information and Protection of Privacy Act (FIPPA), as amended by Bill 194 (in force July 1, 2025), gives individuals the right to request deletion of their personal information. No mechanism existed to fulfill such a request.

What was done: A new authenticated endpoint (DELETE /api/student/<id>/data) allows a professor to anonymize a student's personal information. Upon request:

Name, email address, student number, RFID card ID, and password are overwritten with anonymized placeholder values.

Grade, attendance, poll response, and participation records are preserved — these are academic records required for institutional record-keeping, and are no longer linkable to the individual's identity after anonymization.

The action is logged in the audit trail with professor ID, student ID, and timestamp.

Access control: only a professor who has that student enrolled in one of their classes can trigger the deletion, preventing cross-professor data operations.

Y1 Session Timeout

What was done: Professor sessions now expire automatically after 3 hours of inactivity. Previously, sessions had no expiry — a professor who left a browser open on a shared or unattended device would remain logged in indefinitely.

Y2 Password Policy

What was done: All password fields (professor account creation, professor password reset, and student first-time password setup) now enforce a minimum standard consistent with NIST SP 800-63B:

Minimum 10 characters  |  At least one number  |  At least one symbol

Previously, student passwords required only 6 characters with no complexity requirements, and professor registration had no minimum at all.

Y3 Database Encryption at Rest

What it is: The development database is SQLite — a single file on disk with no encryption. This is not acceptable for production storage of student PII.

Resolution (deployment configuration, not code): Production deployment will use Azure Database for PostgreSQL (Flexible Server) in the Canada Central region. Azure PostgreSQL provides encryption at rest using AES-256, automated backups, and point-in-time restore. The application code requires no changes — SQLAlchemy already supports PostgreSQL via a connection string set in the environment configuration.

Y4 Password Reset Token Disk Storage

What it is: When email (SMTP) was not configured, password reset tokens were written to a plaintext file on the server's local filesystem. If the server were ever compromised, an attacker could use these tokens to take over professor accounts.

What was done: The application now refuses to start in production mode unless MAIL_SERVER is configured. This ensures the file-based fallback is only reachable in local development, never in a deployed environment handling real accounts.

Y5 Controlled Professor Registration

What it is: Professor account registration was open to anyone who could reach the application URL — no affiliation verification was performed.

What was done: A new environment variable, ALLOWED_EMAIL_DOMAINS, restricts registration to specified email domains. For Western deployment this is set to:

ALLOWED_EMAIL_DOMAINS=uwo.ca,schulich.uwo.ca,ivey.ca,ivey.uwo.ca

Any registration attempt using an email address outside these domains is rejected with an error. Additional domains can be added at any time by updating the environment configuration — no code change is required. When the variable is unset (local development), registration remains open for testing convenience.

Y6 Default Development Credentials Removed from Production

What it is: The application startup script automatically created a professor account with the username professor and password password if the database was empty. This ran on the same code path as production startup, meaning a freshly deployed production instance would contain a well-known default account.

What was done: The default account creation is now gated behind a production-mode check. It will not run when PRODUCTION=1 is set in the environment.

E1 Password-Change Endpoint Authentication Fix

What it is: The endpoint used for a student's first-time password setup did not verify that the caller's session token specifically indicated a "needs password" state. A student with a normal active session token could call this endpoint to overwrite their password without knowing their current password.

What was done: The endpoint now reads both values from the session token (student ID and the needs-password flag) and rejects the request if the flag is not set to true. First-time password setup and password change are now properly separated.

E2 File Upload Size Limit

What it is: Quiz and poll question files are uploaded as Excel spreadsheets. No maximum file size was enforced, meaning an oversized or malformed file could exhaust server memory.

What was done: A 10 MB maximum file size is now enforced at the framework level. Requests exceeding this limit are rejected before the file is processed.

Production Environment Configuration

The following environment variables must be set before production deployment. These are stored in a .env file on the server (never committed to source control):

SECRET_KEY=<minimum 32-character random string> PRODUCTION=1 MAIL_SERVER=smtp.office365.com MAIL_PORT=587 MAIL_USERNAME=your@uwo.ca MAIL_PASSWORD=<app password> MAIL_FROM=your@uwo.ca ALLOWED_EMAIL_DOMAINS=uwo.ca,schulich.uwo.ca,ivey.ca,ivey.uwo.ca SQLALCHEMY_DATABASE_URI=postgresql://<user>:<password>@<host>/<db> SOCKETIO_ALLOWED_ORIGINS=https://<your-domain>

Hosting and Data Residency

Target platform: Microsoft Azure — Canada Central region (Toronto)

Rationale: Azure Canada Central ensures all student personal information remains within Canada, satisfying the data residency considerations under FIPPA. Western University already uses Microsoft 365 and Azure services, and Microsoft's Canadian data residency commitments are well-documented and familiar to Western's privacy office. A Privacy Impact Assessment will be completed with Western's Privacy Office prior to the pilot.
